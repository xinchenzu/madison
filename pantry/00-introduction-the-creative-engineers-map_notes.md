## A. Conceptual foundations

### Execution cheapens; judgment becomes scarce

The opening needs to establish the book's governing claim: AI-assisted production reduces the cost of making plausible artifacts, but it does not remove the need to choose the right problem, verify evidence, position the output, and ship responsibly. A controlled GitHub Copilot experiment found developers completed a JavaScript HTTP-server task 55.8% faster with Copilot, making it a useful empirical anchor for the "execution got cheaper" claim. Stack Overflow's 2024 Developer Survey adds adoption context: developers expected AI tools to be integrated into writing code, testing code, and documentation, while misinformation remained a major ethical concern.

**Common misconception:** Students often infer that faster artifact generation means weaker need for planning. The book must invert this: faster generation increases the penalty for unclear goals because bad artifacts can now multiply quickly.

**Worked example:** A student can generate three portfolio pages in an afternoon. The useful question is not "which looks best?" but "which page makes a specific promise to a specific audience and proves it?"

**Source(s):** GitHub Copilot productivity study: https://arxiv.org/abs/2302.06590; Stack Overflow 2024 AI survey: https://survey.stackoverflow.co/2024/ai

### Running project as learning architecture

The introduction should define the running project as the structural spine: a public-facing AI-assisted artifact that becomes a brand system, portfolio object, and launch package. This prevents the book from becoming a survey of tools.

**Common misconception:** Readers may think the running project is an optional exercise. It should be framed as the book's assessment architecture.

**Worked example:** A "personal AI market research assistant" can become a PRD, workflow, agent card, interface, brand strategy, case study, portfolio centerpiece, and pitch deck.

**Source(s):** TIKTOC.md; local repo docs `docs/manuscript.md`, `docs/workflows.md`.

## B. Domain examples and cases

### Case: AI code productivity as signal erosion

The Copilot study gives a concrete story: a task that once signaled technical competence becomes less differentiating when AI assistance compresses completion time. Use this to motivate brand, scope, and proof.

### Case: Stack Overflow's AI adoption and distrust pattern

Stack Overflow's 2024 survey shows broad AI workflow expectations and concern about misinformation. This supports the book's dual claim: AI is normal, but verification remains central.

### Failure case: Fluent artifact without thesis

A portfolio with polished screenshots but no audience, claim, or evidence teaches the introduction's core warning: polish without positioning is not professional presence.

## D. Current state of the field

**Settled:** AI tools accelerate many production tasks; tool adoption is mainstream enough that "I can generate with AI" is not a strong differentiator.

**Contested or emerging:** The long-term labor-market value of technical output under AI assistance remains unsettled. The chapter should present a directional bet, not a settled law.

**Key references:**
1. Peng et al., "The Impact of AI on Developer Productivity," 2023 — empirical productivity anchor.
2. Stack Overflow Developer Survey 2024 — adoption and trust context.
3. TIKTOC.md — local learning architecture.

**Recent developments:** Developer AI adoption and agentic coding tools make supervision, verification, and public proof more important than tool novelty.

## E. Teaching considerations

**Where students get stuck:** They confuse activity with evidence of progress.

**Analogies and framings that work:** "The artifact is the receipt; the brand is the reason the receipt matters."

**Exercises that build the target skill:** Baseline project audit: What artifact will you build, who is it for, what will it prove, and what would make it untrustworthy?
