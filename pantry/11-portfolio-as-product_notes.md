## A. Conceptual foundations

### Portfolio as user journey

A portfolio is a product with audience, entry point, conversion path, proof hierarchy, and maintenance. It should answer: Who is this person/product for? What can they do? What evidence proves it? What should the visitor do next?

**Common misconception:** Portfolio equals gallery. A gallery displays; a product guides.

**Worked example:** A portfolio homepage can move from positioning statement to flagship AI tool case study to proof artifacts to contact.

**Source(s):** TIKTOC.md; local `_lib_Build_...md`.

### Proof over polish

Visual polish helps, but portfolio credibility comes from artifacts, constraints, decision notes, before/after evidence, and links to working demos or source.

**Common misconception:** A beautiful portfolio compensates for unclear proof. It usually amplifies the lack of proof.

**Worked example:** Include "Problem, My role, Constraints, System, Verification, Result" for every case study.

**Source(s):** Local `docs/figures-and-assets.md`; local `_lib_Data_Harness_Your_Numbers.md`.

## B. Domain examples and cases

### Case: Brittany Chiang portfolio

Often cited as a developer portfolio reference because it is clear, focused, and widely forked. Verify current GitHub stars/forks before using numbers.

**Source:** GitHub search/manual verification recommended before manuscript.

### Case: Portfolio case study as product page

Treat the flagship project like a landing page: problem, audience, proof, demo, CTA.

### Failure case: Template portfolio with no differentiation

AI/site builders can produce attractive pages that make the student look interchangeable.

## E. Teaching considerations

**Where students get stuck:** They organize by artifact type instead of visitor decision.

**Analogies and framings that work:** "Your portfolio is a sales page for trust."

**Exercises that build the target skill:** Portfolio map: audience, promise, proof sequence, case study, demo, about, contact, analytics, maintenance plan.
